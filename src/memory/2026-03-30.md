# Learnings — 2026-03-30

## 1. HKJC URL 參數：大小寫敏感

### Fixture Scraper
- `zh-hk` 版本（中文）→ 包含 overseas 賽事（日本、澳洲）
- `en-us` 版本（英文）→ **只有本地賽事**（沙田 + 跑馬地）
- 參數名稱：`calmonth`（不是 `calm`）

### Race Results Scraper
- 參數名稱：**全部小寫** — `racecourse=` + `raceno=`（不是 `Racecourse=` + `RaceNo=`）
- 即使 URL 中顯示大寫，實際请求需要小寫才能正確返回數據

### Lesson
> HKJC URL 的查詢參數**小寫敏感**。用大寫 `RaceNo=` 會導致抓不到任何資料，但 HTTP response 仍然返回 200（空內容）。

## 2. 同一天 HV + ST 的錯誤數據

### 問題
- `races` collection 中，2026-03-04、2026-03-18、2026-03-25 都出現了 HV + ST 兩組記錄
- 兩組venue 的馬匹完全相同（同一場比賽被錯誤複製）

### 根本原因
- 最初的 scraper（2026-03-12）使用了錯誤的 overseas URL，導致同一批 races 同時寫入 HV 和 ST

### 修復
- 2026-03-04：HV 的 9 場是空的，刪除
- 2026-03-18：ST 的 9 場是假的，刪除
- 2026-03-25：ST 的 9 場是假的，刪除

### Lesson
> 當同一場賽事的馬匹出現在不同 venue 時，說明 venue 分類有問題。應該從 HKJC results page 驗證真實 venue。

## 3. 2026-03-29 賽果缺失

### 問題
- `races` collection 中 2026-03-29 完全為空（0 races）
- `racecards` collection 中有 11 場排位表
- Pipeline 的 `sync_past_race_results` 從未成功抓取

### 根本原因
- URL 參數用了大寫（`RaceNo=`）導致抓不到資料
- Pipeline 的 gap analysis 一直報告「數據完整」，因為 `_get_actual_race_count()` 只計數、不寫入
- 結果從未真正執行過抓取

### 修復
- 修正 URL 為小寫 `raceno=`
- 確認 2026-03-29 只有 ST 賽事（Races 2-11，R1 是 S4 日本 overseas）

### Lesson
> `_get_actual_race_count()` 是個「虛報」機制 — 它告訴你 HKJC 網站有多少場，但**不負責抓取**。Pipeline 一直以為數據完整，但實際上 `_scrape_and_sync` 從未成功過。

## 4. Fixtures `race_count` 全部錯誤

### 問題
- 整個 `fixtures` collection 的 `race_count` 都是舊 scraper 的錯誤值
- 例如：2026-03-25 顯示 `race_count=3`，實際是 9

### 根本原因
- overseas URL 解析出來的最高場次數永遠是較小的數字

### 修復
- 用 `races` collection 的實際計數直接覆寫 `race_count`
- 同時補足缺失的 HV fixtures（共補充 200+ 個歷史記錄）

### Lesson
> `race_count` 是個「快取」值，不可靠。Pipeline 每次運行都會重新從 HKJC 抓真實場次，但這個錯誤值已經污染了 `fixtures` collection。

## 6. Docker Container 檔案 vs 本地檔案

- **本地修改**：`src/src/crawler/*.py`（正確的 Python module 結構）
- **容器內修改**：`hkjc-pipeline:/app/src/crawler/`（需要 `docker cp` 同步）
- 兩邊是完全獨立的 filesystem，沒有自動同步

### Lesson
> 容器內的修改**不會自動回寫到本地**。本地修改後需要 `docker cp` 才能生效。Pipeline container 內的 `/app/` 是工作目錄，不是從本地掛載的。
